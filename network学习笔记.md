## 一、网络模型初识
### 1. 网络前世今生
- 网络的核心目的：实现不同设备间的数据传输与资源共享
- 发展脉络：ARPANET → TCP/IP 协议 → 互联网 → 移动互联网 → 万物互联

### 2. 网络模型基本概念
- **OSI 七层模型**：物理层、数据链路层、网络层、传输层、会话层、表示层、应用层
- **TCP/IP 四层模型**：网络接口层、网际层（IP）、传输层（TCP/UDP）、应用层（HTTP/HTTPS）
- 核心思想：分层解耦，每层只关注自己的功能，上层依赖下层提供的服务

## 二、TCP/IP 初识
### 1. TCP 的特点
- **面向连接**：通信前需建立三次握手连接，通信后四次挥手释放连接
- **可靠传输**：通过确认应答、超时重传、流量控制、拥塞控制保证数据不丢包、不乱序
- **面向字节流**：数据以字节流形式传输，无消息边界
- **全双工**：通信双方可同时发送和接收数据

### 2. TCP 报文结构
- 源端口、目的端口：标识通信双方的应用进程
- 序号（SEQ）：标识数据的顺序，解决乱序问题
- 确认号（ACK）：期望收到对方下一个报文段的序号，确认已收到的数据
- 标志位：SYN（建立连接）、ACK（确认）、FIN（释放连接）、RST（重置连接）、PSH（推送数据）、URG（紧急数据）
- 窗口大小：用于流量控制，告知对方自己的接收缓冲区大小

### 3. TCP 建立连接与释放连接的过程
- **三次握手（建立连接）**：
  1. 客户端发送 SYN 报文（SYN=1，SEQ=x），请求建立连接
  2. 服务器回复 SYN+ACK 报文（SYN=1，ACK=x+1，SEQ=y），同意连接
  3. 客户端回复 ACK 报文（ACK=y+1，SEQ=x+1），连接建立成功
- **四次挥手（释放连接）**：
  1. 客户端发送 FIN 报文，请求关闭连接
  2. 服务器回复 ACK 报文，确认收到关闭请求
  3. 服务器发送 FIN 报文，请求关闭服务器到客户端的连接
  4. 客户端回复 ACK 报文，确认收到，连接完全释放

## 三、HTTP 初识
### 1. HTTP 报文结构
- **请求报文**：请求行（方法 + URL + 版本）、请求头、空行、请求体
- **响应报文**：状态行（版本 + 状态码 + 描述）、响应头、空行、响应体

### 2. HTTP 状态码的含义
- 1xx：信息响应（如 100 Continue）
- 2xx：成功响应（如 200 OK、201 Created）
- 3xx：重定向（如 301 永久重定向、302 临时重定向、304 缓存命中）
- 4xx：客户端错误（如 400 错误请求、401 未授权、404 资源不存在）
- 5xx：服务器错误（如 500 服务器内部错误、502 网关错误、503 服务不可用）

### 3. HTTP 请求/响应过程
1. 客户端（浏览器/程序）解析 URL，通过 DNS 解析服务器 IP
2. 客户端与服务器建立 TCP 连接（HTTP/1.1 默认长连接）
3. 客户端发送 HTTP 请求报文
4. 服务器接收请求，处理业务逻辑，生成 HTTP 响应报文
5. 客户端接收响应，解析并渲染/处理数据
6. 连接关闭或复用（根据 Connection 头决定）

### TCP 面向字节流
比喻：自来水管，连续水流
特点：无天然消息边界，只传一串连续字节
现象：操作系统缓冲区合并 / 拆分数据 → 产生粘包、拆包
优势：可靠、有序、不丢包
劣势：需要手动处理消息边界
### UDP 面向数据报
比喻：独立快递包裹，一件一件
特点：一条发送 = 一条接收，天然有消息边界
现象：应用层无粘包、拆包
优势：简单、速度快、无需处理边界
劣势：不保证可靠，可能丢包、乱序
### 固定长度分包
原理：所有消息强制统一长度
优点：简单、稳定、不易错
缺点：浪费带宽、不能发长消息、扩展性差
口诀：统一长度最简单，浪费带宽不灵活
